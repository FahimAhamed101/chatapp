Wellbyn Chat System

A real-time messaging platform with WebSocket support, JWT authentication, and role-based access control. Built with Next.js frontend and Node.js/Express backend.

📋 Project Overview

The Wellbyn Chat System is a modern, full-stack messaging application designed to facilitate real-time communication between two types of users: Users and Senders. The system provides a seamless experience for initiating conversations, sending messages, and maintaining real-time connectivity.

🎯 Purpose

Enable real-time communication between Users and Senders with persistent message history and live status updates.

📱 Tech Stack

Frontend: Next.js 15, React 19, Tailwind CSS, Redux Toolkit
Backend: Node.js, Express.js, TypeScript, MongoDB
Real-time: Socket.io, JWT

🔐 Authentication

JWT-based authentication with role-based access control (RBAC). Secure password hashing with bcrypt. Refresh token mechanism for session management.

ℹ️ Key Information: This is a two-role system - Users and Senders. Each user type has specific features and permissions tailored to their use case.

🔄 System Flow

1. Authentication Flow

User → Register/Login → Backend Validation → JWT Generated → Stored in Redux → Redirect to Inbox

2. Conversation Flow

User Selects Contact → API Call → Get/Create Conversation → Load Messages → Render Chat Window

3. Real-time Messaging Flow

User Sends Message → HTTP POST → Message Saved → Socket Emit → Other User Receives → UI Updates

4. Online Status Flow

Socket Connected → User Status Updated → Broadcast to Other User → Realtime Status Indicator

🔌 Backend REST API

ℹ️ Base URL: All endpoints require authentication header: Authorization: Bearer <accessToken>

Authentication Endpoints

POST /api/auth/register

Description: Create a new user account

Request Body:

{ "email": "user@example.com", "password": "securePassword123", "fullName": "John Doe", "role": "user" // or "sender" }

Response (201):

{ "user": { "_id": "507f1f77bcf86cd799439011", "email": "user@example.com", "fullName": "John Doe", "role": "user" }, "accessToken": "eyJhbGc...", "refreshToken": "eyJhbGc..." }
POST /api/auth/login

Description: Authenticate user and receive tokens

Request Body:

{ "email": "user@example.com", "password": "securePassword123", "role": "user" // or "sender" }

Response (200): Same as register response

Chat Endpoints

POST /api/chat/conversations

Description: Get or create a conversation

Request Body:

{ "recipientId": "507f1f77bcf86cd799439011", "recipientModel": "User" // or "Sender" }

Response (201):

{ "conversationId": "507f1f77bcf86cd799439012" }
GET /api/chat/conversations

Description: List all conversations for authenticated user

Query Parameters:

Parameter Type Description
page number Page number (default: 1)
limit number Items per page (default: 10)

Response (200):

{ "total": 5, "page": 1, "limit": 10, "data": [ { "conversationId": "507f...", "user": { "_id": "...", "name": "John", "email": "..." }, "sender": null, "lastMessageText": "Hello!", "unreadCount": 2 } ] }
GET /api/chat/conversations/:conversationId/messages

Description: Retrieve messages from a conversation

Query Parameters:

Parameter Type Description
limit number Messages per request (default: 20)
before ISO date Load messages before this date

Response (200):

{ "data": [ { "_id": "507f...", "conversationId": "507f...", "senderId": "507f...", "senderRole": "user", "content": "Hello there!", "createdAt": "2024-03-14T10:30:00Z", "readByUser": true, "readBySender": false } ] }
POST /api/chat/conversations/:conversationId/messages

Description: Send a message in a conversation

Request Body:

{ "content": "This is my message!" }

Response (201):

{ "_id": "507f...", "conversationId": "507f...", "senderId": "507f...", "senderRole": "user", "content": "This is my message!", "createdAt": "2024-03-14T10:35:00Z", "readByUser": true, "readBySender": false }
POST /api/chat/conversations/:conversationId/read

Description: Mark all messages in conversation as read

Response (200):

{ "success": true, "message": "Conversation marked as read" }
GET /api/chat/contacts

Description: List all registered users/senders for starting conversations

Query Parameters:

Parameter Type Description
search string Search by name or email

Error Responses

Status Error Code Description
400 VALIDATION_ERROR Request validation failed
401 UNAUTHORIZED Missing or invalid token
403 FORBIDDEN Insufficient permissions
404 NOT_FOUND Resource not found
409 CONFLICT Resource already exists
500 SERVER_ERROR Internal server error

🗄️ Database Schema

User Model (Senders)

{ "_id": ObjectId, "email": String (unique), "password": String (hashed), "fullName": String, "role": "sender" | "user" | "admin", "profilePicture": String (URL), "authProvider": "local" | "google" | "facebook", "isActive": Boolean, "createdAt": Date, "updatedAt": Date }

Doctor Model (Users)

{ "_id": ObjectId, "email": String (unique), "password": String (hashed), "fullName": String, "role": "user", "specialization": String, "licenseNumber": String, "profilePicture": String (URL), "isActive": Boolean, "createdAt": Date, "updatedAt": Date }

Chat Conversation Model

{ "_id": ObjectId, "userId": ObjectId (ref: "Doctor"), "senderId": ObjectId (ref: "User"), "initiatedBy": "user" | "sender", "lastMessage": String, "lastMessageTime": Date, "status": "active" | "archived" | "deleted", "createdAt": Date, "updatedAt": Date, // Index: { "userId": 1, "senderId": 1 } (unique) }

Chat Message Model

{ "_id": ObjectId, "conversationId": ObjectId (ref: "ChatConversation"), "senderId": ObjectId, "senderRole": "user" | "sender", "content": String, "attachments": [ { "type": String, "url": String, "fileName": String } ], "readByUser": Boolean, "readBySender": Boolean, "readAt": Date, "deletedAt": Date, "createdAt": Date, "updatedAt": Date }

Database Indexes

Collection Index Purpose
users { email: 1 } Fast email lookups
conversations { userId: 1, senderId: 1 } Unique conversations
messages { conversationId: 1 } Message retrieval
messages { createdAt: -1 } Chronological sorting

⚡ WebSocket (Socket.io) Events

ℹ️ Connection: Socket.io connects using JWT authentication via token header or Authorization bearer token

Client-to-Server Events

Event: chat:message

Emitted by: Client (when message is sent)

Payload:

{ "conversationId": "507f...", "senderId": "507f...", "senderRole": "user", "content": "Hello!", "timestamp": "2024-03-14T10:35:00Z" }
Event: chat:typing

Emitted by: Client (when user starts typing)

Payload:

{ "conversationId": "507f...", "userName": "John Doe" }

Server-to-Client Events

Event: chat:message

Broadcast to: Other participant in conversation

Payload: Same as above

Event: chat:read

Broadcast to: Message sender

Payload:

{ "conversationId": "507f...", "messageId": "507f...", "readAt": "2024-03-14T10:36:00Z" }
Event: user:online

Broadcast to: All connected users

Payload:

{ "userId": "507f...", "status": "online" }
Event: user:offline

Broadcast to: All connected users

Payload:

{ "userId": "507f...", "status": "offline" }

⚙️ Setup & Installation

Prerequisites

  • Node.js 18+ and npm/yarn
  • MongoDB Atlas account or local MongoDB
  • Git for version control
  • Environment variables configured

Backend Setup

# 1. Navigate to backend directory cd chat-system # 2. Install dependencies npm install # 3. Create .env file cp .env.example .env # 4. Configure environment variables # MONGODB_URI=mongodb+srv://... # JWT_SECRET=your-secret-key # PORT=5000 # 5. Start development server npm run dev # Server runs at http://localhost:5000

Frontend Setup

# 1. Navigate to frontend directory cd wellbyn-chat-ap # 2. Install dependencies npm install # 3. Create .env.local file # NEXT_PUBLIC_API_URL=http://localhost:5000 # NEXT_PUBLIC_SOCKET_URL=http://localhost:5000 # 4. Start development server npm run dev # Frontend runs at http://localhost:3000
⚠️ Important: Make sure both backend and frontend are running for full functionality. Backend on port 5000, Frontend on port 3000.

Environment Variables

Backend (.env)
Variable Description Example
MONGODB_URI MongoDB connection string mongodb+srv://user:pass@cluster.mongodb.net/dbname
PORT Backend server port 5000
ACCESS_TOKEN_SECRET JWT access token secret your-random-secret-key
REFRESH_TOKEN_SECRET JWT refresh token secret your-random-refresh-secret
NODE_ENV Environment mode development | production
Frontend (.env.local)
Variable Description Example
NEXT_PUBLIC_API_URL Backend API base URL http://localhost:5000
NEXT_PUBLIC_SOCKET_URL Socket.io server URL http://localhost:5000

Testing the Setup

# 1. Test backend curl http://localhost:5000/health # 2. Open frontend http://localhost:3000 # 3. Register two accounts (one as User, one as Sender) # 4. Start a conversation between them # 5. Verify real-time messaging works

🚀 Deployment Guide

Backend Deployment (Heroku / Railway)

# 1. Build TypeScript npm run build # 2. Ensure package.json has start script "start": "node dist/server.js" # 3. Deploy to Heroku heroku login heroku create your-app-name git push heroku main # 4. Set environment variables heroku config:set MONGODB_URI=... heroku config:set ACCESS_TOKEN_SECRET=...

Frontend Deployment (Vercel / Netlify)

# 1. Build Next.js npm run build # 2. Deploy to Vercel (recommended for Next.js) npm install -g vercel vercel # 3. Configure environment variables in Vercel dashboard # Add NEXT_PUBLIC_API_URL pointing to deployed backend

Database Deployment

  • Use MongoDB Atlas for managed MongoDB
  • Create a free tier cluster
  • Enable IP whitelist for security
  • Backup regularly using Atlas backup
  • Monitor performance and scaling

Production Checklist

□ Set NODE_ENV=production □ Use environment variables for all secrets □ Enable HTTPS/SSL certificates □ Configure CORS with specific domains □ Set up rate limiting □ Enable database backups □ Configure error logging (Sentry) □ Set up monitoring (DataDog, New Relic) □ Test all endpoints thoroughly □ Load test with multiple users □ Set up CI/CD pipeline □ Document deployment process
